# DSCAN — полное руководство

Beaver Adoption Scanner. CLI-тулза для T-Bank, которая сканирует консьюмер-репо и считает adoption design system'ы **Beaver** против локальных shadow-компонентов.

Этот документ — **единая точка входа для оператора**. Тут установка, запуск, конфиги, формулы метрик, чтение HTML-отчёта и устранение неполадок.

PRD (исходная спецификация) — [`ds-adoption-scanner-prd-v2.md`](../ds-adoption-scanner-prd-v2.md). Архитектура pipeline'а под капотом — [`docs/архитектура.md`](архитектура.md). Каталог warning-кодов — [`docs/предупреждения.md`](предупреждения.md).

## 1. Что делает DSCAN

**Кратко.** Сканирует ~100 React-репо (TypeScript / JSX), классифицирует каждое JSX-использование компонента в один из трёх бакетов:

- **adoption** — Beaver-компонент или локальная обёртка над Beaver без кастомизации UI.
- **shadow** — локальный компонент, дублирующий Beaver (параллельный UI-слой).
- **neither** — утилитарные компоненты: провайдеры, hooks, data-фетчеры, layout-обёртки.

Считает 5 метрик (см. [§7](#7-формулы-метрик)), выдаёт:

- `dataset.jsonl` — instance-level (одна строка = одно JSX-использование).
- `aggregates.json` — derived-метрики A/B/C/D/E + инварианты + warnings.
- `report.html` — self-contained viewer (один файл, открывается в любом браузере без бекенда).
- `warnings.json` — отдельный список диагностик.

**Use case оператора.** Раз в 1–2 недели Stanislav (или будущий on-call) запускает скан, открывает `report.html`, читает метрики, принимает roadmap-решения: какие компоненты добавить в Beaver, в какие репо идти первыми, какие пакеты deprecate.


Команда добавляет ключ на текущую shell-сессию. Без этого `git clone` упадёт с понятной ошибкой — fail-fast, retry'я нет (PRD §8.3).

## 4. Запуск

### Локальный прогон на встроенных фикстурах (без T-Bank доступа)

Полезно для проверки что окружение настроено. `BEAVER_LOCAL_PATH` обходит git clone Beaver-репо и подсовывает фейк из `tests/fixtures/beaver-ui/`:

```bash
BEAVER_LOCAL_PATH=./tests/fixtures/beaver-ui \
  npm run dev -- run --config ./example/.beaver-scan.config.ts
```

Артефакты — в `example/results/`. Открой `report.html` в браузере.

### Pilot-прогон на 5–10 реальных репо

```bash
# 1. SSH-ключ в агент.
ssh-add ~/.ssh/id_ed25519

# 2. Клонировать репо из repositories.json в ./ds-projects/<name>/
node scripts/clone-repos.mjs                  # идемпотентно: пропускает уже склонированное

# 3. Собрать (опционально, dev-режим тоже работает)
npm run build

# 4. Сканирование
npx ds-scanner analyze --output .ds-metrics/report
```

Длительность: 1–5 минут на стандартной машине.

### Полный прогон на 100 репо

```bash
node scripts/clone-repos.mjs
npx ds-scanner analyze --config ds-scanner.config.ts --output .ds-metrics/report
```

Длительность: 10–30 минут (PRD §8.2 SLA target). Если упирается — можно эскалировать в M6b (worker pool, отложенная веха, см. [implementation/plan.md](../implementation/plan.md)).

### Команды CLI

```bash
# Сканирование
ds-scanner analyze [--config <path>] [--output <dir>]   # operator-friendly defaults
ds-scanner run --config <path> [--output <dir>] [--no-fail-on-invariant]

# Постобработка
ds-scanner aggregate --dataset <path> --out <dir>       # пересчёт из существующего dataset.jsonl
ds-scanner viewer --aggregates <path> --out <path>      # сгенерировать HTML из агрегатов

# Кэш
ds-scanner update --config <path>                       # git pull всех кэшированных репо
ds-scanner clean --config <path>                        # удалить .cache/
```

**Дефолты `analyze`:**
- `--config` → `ds-scanner.config.ts`
- `--output` → `.ds-metrics/report`

**Различие `analyze` vs `run`:**
- `analyze` — для оператора. Удобные дефолты.
- `run` — explicit. Output берётся из `config.output.dir`, если `--output` не указан.

### Коды выхода

| Код | Значение |
|-----|----------|
| 0 | Успех. Все артефакты записаны, инварианты выполняются. |
| 1 | Общая фатальная ошибка (сеть, FS, etc.). Stack trace в stderr. |
| 2 | Невалидный конфиг — Zod падение. Stderr содержит путь к проблемному полю. |
| 3 | Нарушение инвариантов (§10.1 PRD). Перезапусти с `--no-fail-on-invariant` чтобы получить артефакты и посмотреть `aggregates.invariants.violations[]`. |

## 5. Конфигурация

### Global config (`ds-scanner.config.ts`)

Создаётся в рабочей директории. По умолчанию `analyze` ищет `./ds-scanner.config.ts`, но можно подсунуть любой путь через `--config`.

Полный пример с дефолтами:

```ts
import { defineConfig } from 'beaver-scan';

export default defineConfig({
  // SSH URL Beaver-репо. Клонируется один раз в .cache/beaver-ui/.
  beaverUrl: 'ssh://git@gitlab.tbank.ru:7999/beaver-ui/beaver-ui.git',

  // Файл со списком консьюмер-репо. Резолвится относительно директории
  // этого конфига.
  repositoriesFile: './repositories.json',

  // Куда писать артефакты. dir резолвится относительно конфига.
  output: {
    dir: './.ds-metrics/report',
    formats: ['jsonl', 'aggregates', 'html'],   // подмножества — допустимы
  },

  // Пороги для классификации и warnings.
  thresholds: {
    reusableLocalFiles: 2,             // §5.1: reusable-local срабатывает на 2+ файлах
    substantialMarkupElements: 5,      // §5.1: substantial-markup на 5+ JSX элементов
    unresolvedDynamicWarningPct: 0.05, // §5.4: warning когда unresolved > 5%
    shadowFalsePositiveTarget: 0.15,   // §1.5: целевой FP-rate (для документации, не enforce)
    codeSnippetMaxLines: 200,          // §6.1: usercode сниппет в ShadowComponentRecord
  },

  // Stage 7. Отключается per-repo если конкретное репо не на React Router v6.
  routeResolution: {
    enabled: true,
    router: 'react-router-v6',
    propagationBoundary: 'repo',           // не выходим за границы репо при построении import-graph
    importGraphDepthLimit: 20,
    collapseNestedRoutes: true,            // <Routes> внутри фич схлопываются к родительскому
  },

  // Override whitelist primitive-имён §5.2 (опционально).
  // primitiveNames: ['Button', 'Input', 'Card', ...],

  // ⭐ Cross-repo design kits.
  // Объяви один раз — применится ко всем консьюмерам.
  // source.path резолвится относительно директории этого конфига.
  sharedLibraries: [
    {
      libId: 'team-platform',
      matchPattern: '@team/platform',
      source: { type: 'local-path', path: './shared-kits/team-platform' },
      kind: 'partially-beaver-backed',
    },
  ],
});
```

### `repositories.json`

```json
[
  { "name": "consumer-app-1", "gitUrl": "ssh://git@gitlab.tbank.ru:7999/consumer/app-1.git" },
  { "name": "consumer-app-2", "gitUrl": "ssh://git@gitlab.tbank.ru:7999/consumer/app-2.git" }
]
```

Поля:
- `name` (опционально) — алиас. Если пусто — выводится из `gitUrl`.
- `gitUrl` — SSH URL.
- `localPath` (опционально) — путь к уже склонированному репо. Обходит git clone.
- `config` (опционально) — inline per-repo override. См. ниже.

### Per-repo config — три уровня precedence

Консьюмер-репо **не обязаны** добавлять `.beaver-scan.json` — работают built-in defaults (PRD §9.2 разворот: исходно файл был required, но pilot-feedback показал что координировать 100 команд нецелесообразно).

Приоритет (от высшего к низшему):

1. **`.beaver-scan.json` у консьюмера**, лежит в корне его репо. Опт-ин. Если есть — побеждает.
2. **Inline `config`** на записи репо в `repositories.json`. Оператор использует когда хочет переопределить дефолт без правки консьюмера:
   ```json
   {
     "name": "consumer-app-1",
     "gitUrl": "ssh://...",
     "config": {
       "exclude": ["src/legacy/**"],
       "localLibraries": [
         {
           "libId": "team-ui-kit",
           "matchPattern": "src/shared/ui-kit/**",
           "source": { "type": "local-path", "path": "src/shared/ui-kit" },
           "kind": "partially-beaver-backed"
         }
       ]
     }
   }
   ```
3. **Built-in defaults**:
   - `include: ['src/**/*.{ts,tsx,js,jsx}']`
   - `exclude: []` (но `DEFAULT_EXCLUDES` всегда применяется — см. ниже)
   - `tsconfig: 'tsconfig.json'`
   - `localLibraries: []`
   - `routeResolution: enabled` (если в global enabled)

### Что **всегда** исключается из скана

`DEFAULT_EXCLUDES` в [`src/config/schema.ts`](../src/config/schema.ts):

```
**/node_modules/**
**/dist/**
**/build/**
**/.next/**
**/*.test.*
**/*.spec.*
**/*.stories.*
**/__mocks__/**
**/__fixtures__/**
**/mocks/**
**/*.mock.*
**/*.fixture.*
```

Чтобы пере-включить (специфический use case — например, нужно сканировать `*.stories.*`) — пропиши `exclude: []` в per-repo конфиге и явно `include` нужные паттерны.

### Поля `LocalLibrary` (для `sharedLibraries` и per-repo `localLibraries`)

```ts
{
  libId: string;             // уникальный ID — используется для override (per-repo поверх shared)
  matchPattern: string;      // picomatch glob ИЛИ имя npm-пакета (@team/kit)
  source: {
    type: 'local-path';
    path: string;            // путь к директории-источнику. Резолвится:
                             //   - для sharedLibraries: относительно configDir
                             //   - для localLibraries: относительно репо
  };
  kind: 'partially-beaver-backed' | 'fully-custom';
}
```

`kind` — fallback на случай, когда prescan не сможет распарсить источник (например, npm-пакет без исходников). Если prescan успешен, использует данные оттуда per-component.

**Конфликт `libId`** между shared и per-repo: per-repo **полностью заменяет** shared (entry-level replacement, не field-level merge). Так удобно делать точечные исключения.

### Невалидный конфиг

- Битый JSON / Zod failure: exit 2, в stderr — путь к проблемному полю.
- Отсутствие `.beaver-scan.json` у консьюмера: норма, применяются defaults.

## 7. Формулы метрик

Все формулы детерминированы. PRD §7.

### Бакеты

```
adoption   = usage с bucket=adoption
shadow     = usage с bucket=shadow
neither    = usage с bucket=neither (не в знаменатель A/B/E)
```

### A — Global Adoption Rate

```
A = Σ(bucket=adoption) / (Σ(bucket=adoption) + Σ(bucket=shadow))
```

Считается по **всем** сканированным репо. `neither` **не** в знаменателе. Если adoption+shadow = 0 → A = 0 (не NaN).

### B — Per-Repo Adoption Rate

Та же формула, но для каждого `repoId` отдельно:

```
B(r) = Σ(bucket=adoption ∧ repoId=r) / (Σ(bucket=adoption ∧ repoId=r) + Σ(bucket=shadow ∧ repoId=r))
```

В viewer'е дополнительно вычисляется hero-card «Среднее по репо» — невзвешенное среднее `B(r)` по всем r.

### C — Shadow Landscape (Migration Backlog)

Не формула, а **два датасета**:

**C.1 — Per-component (основной).** Агрегация по `groupKey = sha256(componentName + sorted(propNames) + jsxElementCount_bucket)`. Одна запись = одна группа shadow-компонентов через все репо.

Hero-card «Confirmed shadow» считает количество групп с уровнем `confirmed`.

**C.2 — Per-file.** Одна запись = один shadow-файл в конкретном репо. Для конкретных миграционных задач.

### D — Beaver Package Coverage

Для каждого Beaver-пакета:
- `reposUsing` — в скольких из сканированных репо импортируется хоть раз.
- `instances` — общее число JSX-использований (после канонизации через агрегатор).

### E — Per-Route Adoption (PRD §7.5)

```
E(r, p) = Σ(bucket=adoption ∧ route.kind=bound ∧ route.path=p ∧ repoId=r) /
          (Σ(bucket ∈ {adoption, shadow} ∧ route.kind=bound ∧ route.path=p ∧ repoId=r))
```

**В знаменатель идут ТОЛЬКО `route.kind === 'bound'`.**

- `shared` (файл достижим от 2+ маршрутов) — **не** идёт в E.
- `unmapped` (не достижим от ни одного маршрута: app shell, провайдеры, utils) — **не** идёт в E.
- `unsupported` (репо без React Router v6) — не даёт записей в E.

Sharеd-компоненты трекаются отдельно — `aggregates.metrics.sharedComponentsAdoption`. Их adoption важен для стратегии, но они не принадлежат одному роуту.

### Что **не** делает сканер

Traffic-aware weighting на метрику E (взвешивание по бизнес-важности роута) — **не** делается. BI применяет веса поверх E.

## 8. Чтение HTML-отчёта

`report.html` — self-contained: один файл, inline CSS+JS, данные через `const DATA = ...`. Открывается в любом браузере без backend'а и без интернета.

### Layout

```
┌────────────────────────────────────────────────────────┐
│  BEAVER · Adoption Scanner          Сканирование от .. │
├────────────────────────────────────────────────────────┤
│  Hero: A · B · C · D                                   │
├────────────────────────────────────────────────────────┤
│  E — Adoption по маршрутам                             │
│    [фильтр] [строк на странице]                        │
│    | репо | маршрут | adoption | adoption/shadow |     │
│    [pager]                                             │
├────────────────────────────────────────────────────────┤
│  B — Adoption по репо                                  │
├────────────────────────────────────────────────────────┤
│  C — Карта shadow-компонентов                          │
│    [По компонентам] [По файлам]                        │
├────────────────────────────────────────────────────────┤
│  D — Покрытие Beaver-пакетов                           │
├────────────────────────────────────────────────────────┤
│  Shared-компоненты                                     │
├────────────────────────────────────────────────────────┤
│  Предупреждения и инварианты [Показать]                │
└────────────────────────────────────────────────────────┘
```

### Контролы в каждой таблице

- **Фильтр** — поиск по всем строковым колонкам (substring, case-insensitive, debounced 150ms). Цифры «10 из 7127» — отфильтровано из общего.
- **Строк на странице** — 50 / 100 / 250 / 500 / «все». Дефолт 50.
- **Сортировка** — клик по заголовку колонки. Цикл: ↑ → ↓ → выкл. Стрелка ▲/▼ показывает текущее.
- **Пагинация** — `«` `←` `1 / 10` `→` `»` + индикатор «1–50 из 7127».
- **Длинные пути** обрезаются `…` — наведи курсор для полного значения.

### Как читать каждую секцию

**Hero** — пульс-чек: общий adoption, разброс по репо (среднее B), сколько shadow-групп найдено, сколько пакетов используется.

**E — Adoption по маршрутам.** Используй для **триажа**: «`/checkout` хуже, чем `/admin`» проще понять чем «adoption 67%». Кликни сортировку по `Adoption` ascending — увидишь самые плохие маршруты сверху.

**B — Adoption по репо.** Кто требует внимания. Sort by Adoption asc — приоритезация outreach.

**C — Shadow landscape**:
- **По компонентам** (дефолт) — для стратегических решений «какой компонент добавить в Beaver». Sort by «Всего usage'ей» — топ компонентов, переизобретённых командами.
- **По файлам** — для конкретных миграционных задач после решения «мигрируем X в репо Y».
- **Уровни** (badge): `confirmed` (красный) — почти точно дубль Beaver. `likely` (жёлтый) — высокая вероятность. `possible` (серый) — требует ручного review.

**D — Покрытие Beaver-пакетов.** Маленький `Репо использует` — пакет недопромоушен. Маленький `Использований` — пакет используется редко, может deprecate.

**Shared-компоненты.** Файлы, достижимые от 2+ pages. В метрику E **не идут**. Покажет, какие компоненты живут «на стыке» страниц (хедеры, общие layout'ы) — стратегически важны.

**Предупреждения и инварианты.** Сворачивается по умолчанию. Если что-то найдено — кнопка показывает кол-во. Развернуть → таблица с фильтром/сортировкой по коду / репо / файлу / сообщению.

## 9. JSONL-датасет (поле за полем)

`dataset.jsonl` — формат JSON Lines (одна запись = одна строка, без обрамляющих скобок). Файл может быть тысячи строк — это нормально.

**Три типа записей:**

### `kind: 'usage'` — основное

Одно JSX-использование компонента (`<Foo />`).

```ts
{
  schemaVersion: '1.1',
  kind: 'usage',
  repoId: 'consumer-app-1',
  filePath: 'src/pages/Checkout.tsx',   // repo-relative, forward-slashed
  line: 42,
  column: 6,
  componentName: 'Button',               // или 'Form.Item' для member expressions
  category: 'beaver' | 'local' | 'local-library' | 'third-party' | 'html-native',
  bucket: 'adoption' | 'shadow' | 'neither',
  shadowLevel?: 'confirmed' | 'likely' | 'possible',  // только если bucket=shadow
  classificationSource: 'direct-beaver' | 'beaver-backed-wrapper' |
                        'beaver-composition' | 'wraps-with-customization' |
                        'parallel-local-ui' | 'utility-heuristic' |
                        'unresolved-dynamic',
  // Beaver-specific:
  beaverPackage?: '@beaver-ui/button',
  canonicalizedVia?: '@beaver-ui/components',   // если импорт через агрегатор
  // local-library:
  localLibId?: 'team-ui-kit',
  beaverBackedByLib?: boolean,
  // route binding:
  route:
    | { kind: 'bound'; path: '/checkout' }
    | { kind: 'shared'; paths: ['/a', '/b'] }
    | { kind: 'unmapped' }
    | { kind: 'unsupported' },
  resolution: 'static' | 'dynamic-branch',
}
```

### `kind: 'shadow-component'` — характеристика shadow-компонента

Эмитится один раз на каждый distinct shadow-компонент (`profile.absPath + componentName`). Для AI-слоя (v2).

```ts
{
  schemaVersion: '1.1',
  kind: 'shadow-component',
  repoId, filePath, componentName,
  signature: {
    propNames: ['onClick', 'children', 'variant'],
    jsxElementCount: 7,
    localImports: ['Icon'],
    beaverImports: [],
    htmlTags: ['button', 'span'],
    usesStyled: false,
  },
  pathHint: {
    directorySegments: ['src', 'features', 'checkout', 'ui'],
    feature: 'checkout',
  },
  codeSnippet: '...',          // первые N строк исходника (N из codeSnippetMaxLines)
  codeSnippetTruncated: boolean,
  beaverCandidateMapping: {    // ⏳ заполняется внешним AI-процессом (v2)
    candidatePackage: null,
    candidateComponent: null,
    confidence: null,
    source: null,
  },
  embedding: null,             // ⏳ v2
  usageCount: 7,
  filesUsedIn: 3,
  shadowLevel: 'confirmed' | 'likely' | 'possible',
  signals: ['primitive-like-name', 'substantial-markup', ...],
}
```

### `kind: 'unresolved-dynamic'` — нерезолвимое JSX

Эмитится для динамических конструкций, которые сканер честно не разбирает.

```ts
{
  schemaVersion: '1.1',
  kind: 'unresolved-dynamic',
  repoId, filePath, line,
  reason: 'lookup-by-string' | 'member-expression-not-supported' |
          'external-hoc' | 'spread-component' |
          'nested-ternary' | 'if-else-branch' | 'switch-branch',
  context: '<components[variant]/>',   // короткий сниппет источника
}
```

Если доля `unresolved` превышает `thresholds.unresolvedDynamicWarningPct` (5% по умолчанию) — в `warnings.json` падает `unresolved-dynamic-rate-exceeded`.

### Детерминизм

Записи в `dataset.jsonl` отсортированы по `(repoId, filePath, line, column)` (PRD §8.4). Два прогона на одном SHA дают **байт-идентичный** файл — это инвариант, тест есть.

## 10. Инварианты

Из PRD §10.1 реализовано 7 из 9. Живут в `aggregates.json#invariants.violations[]`. Нарушение → exit 3 (отключается флагом `--no-fail-on-invariant`).

| Код | Что проверяет |
|-----|---------------|
| `shadow-level-consistency` | `bucket=shadow` ⇔ `shadowLevel` присутствует. |
| `beaver-package-canonicalization` | `category=beaver` → `beaverPackage` обязателен. |

| `schema-version-fixed` | Все записи с одним `schemaVersion`. |

| `dataset-completeness-shadow` | `Σ(shadowLandscape.byFile.usageCount) === count(bucket=shadow)`. |

**Не реализовано:**
- #8 per-route denominator correctness — гарантируется конструкцией `computePerRouteAdoption`.
- #9 determinism hash — проверяется на CI vitest-спецификой, не runtime'ом.

- `dataset-completeness-shadow` с дельтой > 0 — почти всегда означает barrel/aliasing-баг в сканере. См. PF2.7 в [implementation/plan.md](../implementation/plan.md). Если на актуальной версии скана — сообщи дельту и `warnings.json`, скорее всего пятый кейс.
- `shadow-level-consistency` / `beaver-package-canonicalization` — баг в классификаторе. Шли репро.
- Остальные — скорее всего код-баг. Перезапусти с `--no-fail-on-invariant`, посмотри список violations, отправь репо в issue.

## 11. Классификация: как сканер решает что shadow, что adoption

### Этапы (PRD §4.6)

**Этап A — структурная категория → bucket:**

| `category` | Default bucket | classificationSource |
|------------|---------------|----------------------|
| `html-native` | `neither` | `utility-heuristic` |
| `beaver` | `adoption` | `direct-beaver` |
| `local-library` с `backedByBeaver=true` | `adoption` | `beaver-backed-wrapper` |
| `local-library` с `backedByBeaver=false` | → Этап B как `fully-custom-lib` | — |
| `third-party` | `neither` | `utility-heuristic` |
| `local` | → Этап B | — |

**Этап B — для `local` и fully-custom local-library.** Это где живёт реальная shadow-детекция.

Порядок применения (без перезаписей):

1. **Neither-эвристики** (PRD §5.3) — применяются **первыми**.
   - `*Provider` / `*Context` / `*Gate` → neither/provider-context
   - `use*` → neither/hook-like (это не компонент, но защита от ошибок)
   - `*Query` / `*Mutation` / `*Fetcher` / `*Loader` / `*Data` (markup ≤ 2) → neither/data-component
   - `*Layout` / `*Page` / `*Template` / `*Shell` / `*Scaffold` (markup ≤ 3) → neither/layout-wrapper
   - `*Container` / `*Wrapper` → neither/business-container

2. **Adoption-wrapper rules (§3.6).** Если компонент **импортирует Beaver**:
   - Передаёт `className` / `style` / `css` / `styled(BeaverX)` → **shadow** / `wraps-with-customization` / `possible`
   - Markup ≥ 5 элементов → **shadow** (substantial)
   - Иначе (композиция без кастомизации) → **adoption** / `beaver-composition`

3. **Shadow-сигналы (§5.1).** Если **нет** Beaver-импортов:

| Сигнал | Правило |
|--------|---------|
| `wraps-with-customization` | className/style на Beaver или `styled(BeaverX)` |
| `standalone-styled` | `styled.button` / `@emotion/styled` без Beaver |
| `primitive-like-name` | имя в whitelist §5.2 |
| `substantial-markup` | ≥ 5 JSX элементов в body |
| `reusable-local` | используется в ≥ 2 файлах |
| `parallel-layer` | лежит в `ui/`, `components/ui/`, `shared/ui/`, `kit/` |
| `multi-route` | компонент достижим от 2+ маршрутов |

4. **Уровни (§3.5).** Если хотя бы один shadow-сигнал сработал:
   - `confirmed`: нет Beaver-импортов + `primitive-like-name` + `substantial-markup`
   - `likely`: нет Beaver-импортов + (`reusable-local` ИЛИ `multi-route`), не confirmed
   - `possible`: иначе

5. **Default**: `neither` / `utility-heuristic`.

### Whitelist primitive-имён (§5.2)

```
Button, Input, TextField, Select, Checkbox, Radio, Switch, Toggle,
Text, Heading, Title, Label, Icon, Avatar, Badge, Tag, Chip, Card,
Modal, Dialog, Drawer, Tooltip, Popover, Popup, Menu, Dropdown,
Tab, Tabs, Panel, Accordion, Divider, Spacer, Stack, Flex, Grid, Box,
Container, Alert, Notification, Toast, Skeleton, Spinner, Loader, Progress
```

Переопределяется per-repo через `primitiveNamesOverride: string[]` (для редких случаев — например, `Card` это домен-карточка платежа, а не UI).

### Пуристский подход к className/style

**Любая** передача `className` или `style` в Beaver-компонент → shadow, даже если type-сигнатура Beaver-компонента это разрешает. Логика: если Beaver не предоставляет нужного пропса, это **бэклог Beaver-команды**, не повод обходить через стилизацию.

Это сознательная продуктовая позиция (PRD §3.6). False-positives ловятся в pilot FP-review и фиксятся через тюнинг whitelist'а / порогов.

## 12. Резолвер маршрутов

Stage 7 (PRD §4.7). Привязывает каждое usage к маршруту через **статический анализ import-графа** от page-компонентов.

### Поддерживаемые формы route-конфига

- `createBrowserRouter([{ path, element|lazy|Component|children: ... }])`
- `createHashRouter([...])`
- `createMemoryRouter([...])`
- Именованный export массива `RouteObject[]`

### Поддерживаемые формы `element`

- `element: <Foo />` → Foo резолвится через ImportMap
- `element: <Pages.Dashboard />` — member expression, depth ≤ 5 (PF2.3 + PF2.6 chain)
- `element: <Guard><Page /></Guard>` — обёртка вокруг page, BFS depth ≤ 5 (PF2.5), первый локальный кандидат побеждает
- `element: <ErrorBoundary><Guard><Layout><Pages.X/></Layout></Guard></ErrorBoundary>` — глубокая обёртка с member-expression child
- `Component: Foo` (RR v6.4+ data-router синтаксис)
- `lazy: () => import('./foo')` — следование default export

### Поддерживаемые формы `path`

- String literal: `path: '/checkout'`
- Identifier const: `path: ROOT_PATH`
- Single member: `path: ROUTER_PATHS.checkout`
- Nested member (depth ≤ 5): `path: ROUTER_PATHS.checkout.payment`
- Cross-file import: `import { ROUTER_PATHS } from '@/constants'`
- TS-обёртки: `as const`, `satisfies T`, `<T>x`, `x!` — strip-аются автоматически
- Template literal без интерполяции: `` path: `/foo` ``

### Что **не** поддерживается

- Template literal с интерполяцией: `` path: `${BASE}/foo` `` → `dynamic-path-template-literal`
- Computed property access: `path: PATHS[key]` → `dynamic-path-non-static-member`
- Function calls: `path: getPath()` → `dynamic-path-skipped`
- Spread paths: `path: [...base, ...feature]` (variable following depth 10) — частично, see PRD §4.7.2
- Сложные conditional element: `element: cond ? <A/> : <B/>` (branch enumeration на element — v2)

Все они эмитятся в `warnings.json` с конкретным под-кодом — открой и читай.

### Route kind на каждом usage

```
'bound'      — файл достижим ровно от одного page-компонента
'shared'     — файл достижим от 2+ page-компонентов
'unmapped'   — файл не достижим от ни одного page-компонента
'unsupported' — репо без поддерживаемого роутера (или Stage 7 disabled)
```

Метрика **E считается только по `bound`**. См. [§7.5](#e--per-route-adoption-prd-75).

### Барrel-chain (PF2.6)

`<Pages.Dashboard/>` через `import * as Pages from './pages'`, где `./pages/index.ts` re-exports из `./pages/Dashboard.tsx` — резолвер автоматически проходит по re-export цепочке (depth ≤ 3) и привязывает route к defining-файлу, а не к barrel'у. Это даёт точное per-route adoption: разные `<Pages.X/>` в роутах биндятся к разным файлам.

## 13. Измерение false-positive rate

PRD §10.3. **Цель MVP: FP-rate ≤ 15%** на ручной выборке из 50 shadow-компонентов.

```bash
# 1. Извлечь 50 случайных shadow-записей
grep '"bucket":"shadow"' .ds-metrics/report/dataset.jsonl | shuf -n 50 > fp-review.jsonl
```

2. Для каждой записи открой `filePath:line` в IDE. Ручной вердикт:
   - **Shadow** (правильно): да, это дубль Beaver-компонента или параллельный UI.
   - **Not shadow** (FP): это утилита / провайдер / специфическая бизнес-логика, а не UI-дубликат.

3. Запиши вердикты в таблицу.

4. **`FP-rate = count(verdict='not shadow') / 50`**.

**Если FP > 15% — тюнинг:**

| Симптом | Что менять |
|---------|-----------|
| Домен-`Card` (карточка платежа) попадает в shadow | `primitiveNamesOverride` в per-repo config — исключить `Card` |
| Мелкие layout-обёртки → shadow | Поднять `thresholds.substantialMarkupElements` с 5 до 7/10 |
| Vendored UI-кит идёт как shadow | Описать как `kind: partially-beaver-backed` в `sharedLibraries` |
| Слишком много confirmed | Сузить whitelist primitive-имён |

Цикл: тюнинг → перезапуск → новая выборка → FP-rate.

## 14. Устранение неполадок

| Симптом | Вероятная причина | Что делать |
|---------|-------------------|------------|
| **Exit 2** | Невалидный конфиг (Zod failure) | stderr содержит путь к проблемному полю. Проверь типы. |
| **Exit 3** | Нарушение инварианта | Перезапусти с `--no-fail-on-invariant` и посмотри `aggregates.invariants.violations[]`. |
| **`dataset-completeness-shadow` violation, дельта > 0** | (Pre-PF2.7) Barrel-aliasing / default-import / `export *`. Должно быть фиксано в `bd0c517`. | Подтяни последний main. Если осталось — пришли `warnings.json` + текст нарушения, возможно пятый паттерн. |
| **`git clone failed` на консьюмере** | SSH / сеть / нет доступа | Stderr — raw git. Retry'я нет (fail-fast). SSH? `ssh-add ~/.ssh/id_ed25519`. |
| **`Invalid per-repo config`** | Малформенный `.beaver-scan.json` или inline override | Stderr показывает Zod-style путь к полю. Отсутствующий конфиг — норма. |
| **`parse-failed` шторм на `.ts` файлах** | (Pre-pilot) Парсер был с `jsx: true` для всех файлов. Фикс в `0e0a4b3`. | Подтяни последний main. JSX теперь enabled только для `.tsx`/`.jsx`. |
| **`unresolved-dynamic-rate-exceeded`** | В репо много `React.createElement` / lookup-by-string | Не фатально — артефакты создаются. Открой `warnings.json` для конкретики. Если паттерн легитимный — подними `unresolvedDynamicWarningPct`. |
| **`route-resolution-warning` с `page-import-not-in-repo`** | Route element ссылается на символ, резолвящийся вне репо | Warning содержит `filePath` + `absPath` route-конфига. Скорее всего, page-компонент импортирован как re-export из external пакета. |
| **`dynamic-path-skipped` на маршруте** | Path использует неподдерживаемую форму: template literal с интерполяцией, computed access, function call | См. [§12](#12-резолвер-маршрутов) — список поддерживаемых форм. Переведи на статический const где возможно. |
| **`jsx-member-unresolved` для `<NS.X/>`** | `NS` не импортирован как in-repo binding, ИЛИ `X` не достижим через резолвленный файл (depth > 3) | Проверь, что `NS` — relative/aliased import (не external). Проверь, что source-файл реально владеет `X`. |
| **Wrapped route element не биндит page** | (Pre-`0e0a4b3`) Обёртка `<Guard><Page/></Guard>` блокировала внутреннее JSX | Подтяни main. Резолвер ходит по JSX-дереву depth 5, первый локальный кандидат побеждает. |
| **`<Form.Item/>` → `unresolved-dynamic`** | (Pre-pilot) Member expressions отбрасывались | Подтяни main. §5.5 канонизирует к base Beaver-пакету. |
| **Shared design kit не подхватился в нескольких репо** | Оператор не объявил | Добавь `sharedLibraries` в global config. `source.path` — относительно конфига. |
| **`local-lib-prescan-failed`** | `source.path` директория отсутствует / не читается | Проверь путь в `localLibraries[].source.path` (или `sharedLibraries[]`). Компоненты из этой либы фолбэкаются на config `kind`. |
| **Много distinct маршрутов через один barrel → все `shared`** | (Pre-`e730db2`) Page резолвился на barrel, не на leaf | Подтяни main. Barrel chain ходит до defining файлов (depth 3). |
| **Кэшированный клон отстал от удалённого** | Drift коммит-timeline | `npx ds-scanner update --config ds-scanner.config.ts` — `git pull --ff-only` по всему кэшу. |
| **Кэш разросся** | Weekly-run default | `npx ds-scanner clean --config ds-scanner.config.ts` — снести `.cache/`. Следующий run клонирует с нуля. |
| **HTML-отчёт огромный, тормозит браузер** | (Pre-PF3.2) Все строки рендерились разом. Фикс: pagination + filter + sort. | Подтяни main. Дефолтный pageSize = 50, можно через select переключить до «все». |

## 15. Когда менять `schemaVersion`

PRD §6.4. Сейчас все записи несут `"schemaVersion": "1.1"`.

| Изменение | Bump |
|-----------|------|
| Добавил optional поле | Patch-совместимо. Не бампить. |
| Добавил новый warning-код в `WARNING_CODES` | Patch-совместимо. Не бампить. |
| Переименовал поле / поменял тип / удалил поле | **Breaking**. Bump до `2.0`. Согласовать с BI до релиза. |
| Изменил формулу метрики (новая интерпретация бакетов) | **Breaking** на уровне семантики. Bump до `2.0`. |

Bump делается в `src/types/dataset.ts` (`SCHEMA_VERSION`). Все эмиттеры подтягивают константу — менять в одном месте.
