## 1. Pipeline

8 стадий из PRD §4. Каждая принимает и отдаёт документированный контракт. Без guard'ов между стадиями — разделение через explicit input/output.

### Граничные правила

- **Stages 1 → 8 — синхронный per-repo цикл** (M6b deferred: worker pool).
- **Beaver prescan — один раз на скан**, ДО repo-loop'а.
- **Local-lib prescan — per-repo** (плюс shared один раз перед loop'ом, merged in loop).
- **Stage 7 опционален** — `routeResolution.enabled` в global или per-repo конфиге.
- **Aggregate — финальная стадия**, не per-repo.

## 3. Типы данных

### `UsageRecord`

Одно JSX-использование. Подробно — [`docs/руководство.md`§9](руководство.md#9-jsonl-датасет-поле-за-полем).

Инвариант: один usage → одна запись. **Нет агрегации** на этом уровне.

### `ShadowComponentRecord`

Одна запись на distinct shadow-компонент. Ключ дедупа — `(absPath, componentName)` после aliasing (PF2.7).

Содержит `signature` для будущего AI-слоя (embedding + Beaver-candidate mapping).

### `UnresolvedRecord`

Для динамических конструкций (`React.createElement(name)`, lookup-by-string, etc.). Не идут в метрики, трекаются отдельно.

### `Aggregates`

```ts
{
  schemaVersion: '1.1',
  meta: { scannerVersion, scannedAt, scanDurationMs, beaverVersion, reposScanned, filesScanned },
  metrics: {
    globalAdoption: { value: number; formula: string },
    perRepoAdoption: Array<{ repoId, value }>,
    shadowLandscape: {
      byFile: Array<ShadowByFile>,
      byComponent: Array<ShadowByComponent>,
    },
    beaverCoverage: Array<{ package, reposUsing, instances }>,
    perRouteAdoption: Array<{ repoId, routePath, value, adoptionInstances, shadowInstances }>,
    sharedComponentsAdoption: Array<{ repoId, filePath, componentName, sharedAcrossRoutes, bucket }>,
  },
  invariants: { checked: number; failed: number; violations: [...] },
  warnings: Warning[],
}
```

### `BeaverRegistry`

Output Stage 5a. Содержит:
- `version` — git describe (тег или SHA)
- `packages: Set<string>` — все обнаруженные `@beaver-ui/*` имена
- `exports: Map<package, Set<symbol>>` — собственные exports пакетов
- `reExports: Map<aggregator, Map<symbol, { sourcePackage, sourceSymbol, hops }>>` — re-export map с депт=5 после flatten'а

### `LocalLibRegistry`

Output Stage 5b. `byLib: Map<libId, Map<componentName, boolean>>` где boolean = `beaverBackedBy`. Плюс `prescanFailed: string[]`.

### `ResolvedLibrary`

Внутренний тип после path-resolution. `ResolvedLibrary { libId, matchPattern, sourceAbsPath, kind, scope: 'shared'|'repo' }`. Используется в `mergeLibraries` для precedence-resolution.

## 4. Детерминизм

PRD §8.4. **Инвариант проекта.** Тест `tests/pipeline.test.ts` describe `determinism (§8.4)`.

### Что обеспечивается

- Записи в `dataset.jsonl` отсортированы по `(repoId, filePath, line, column, kindRank)` через `sortRecords` в [`writer/jsonl.ts`](../src/writer/jsonl.ts).
- Map итерация не используется для пользовательского output (использовать `[...map.entries()].sort()`).
- Wall-clock timestamping только в `meta.scannedAt` (не идёт в hash инвариант).

### Что **не** обеспечивается (но мы заявляем)

- `meta.scannedAt` и `meta.scanDurationMs` меняются между прогонами. **Не учитываются** в determinism-check.
- `aggregates.json` форматируется с pretty-print (2-space indent). JSON.stringify итерирует ключи в insertion-order; объекты собираются детерминированно.

### Что ломает детерминизм (если придёт PR)

- Использование `Array.sort` без явного comparator: для number'ов корректно, для string'ов — тоже, но если в массиве смешанные типы или sparse — UB.
- `Map.forEach` / `for ... of map.entries()` без последующей сортировки.
- `Date.now()` или `process.hrtime` в hot path.
- Чтение файлов в параллельных Promise.all (если не отсортировано на входе).

## 5. Stage 5a: Beaver prescan

[`src/prescan/beaver.ts`](../src/prescan/beaver.ts).

### Что делает

1. **Получает checkout** Beaver-репо. Источники в порядке приоритета:
   - `BEAVER_LOCAL_PATH` env (для тестов / CI)
   - Уже склонированный `.cache/beaver-ui/` (cache hit)
   - `git clone --depth=1 --single-branch` (fail-fast, без retry)

2. **Version**: `git describe --tags --always`. Тег предпочт, fallback на SHA.

3. **Обнаружение пакетов**: `packages/*/package.json` → набор имён.

4. **Парсинг entry-файлов**: `packages/<pkg>/src/index.{ts,tsx,js}`. Извлекаются:
   - `ownExports` (FunctionDeclaration, VariableDeclaration, ClassDeclaration)
   - `reExports` через `export { X } from 'source'`, `export * from 'source'`

5. **Flatten цепочек**: re-export map строится через `traceSymbol` (depth ≤ 5, cycle detection через visited-set).

### Output

`BeaverRegistry`. Передаётся в `collect.ts` и `local-lib.ts` для классификации.

### Тонкости

- **`export * as NS from 'src'`** — namespace export, регистрируется как ownExport (NS), не fan-out.
- **Self-re-exports** (`export { Button } from './Button'` внутри `@beaver-ui/button`) **не считаются re-export'ом**, а становятся ownExport — иначе trace бесконечно ходит сам в себя.

## 6. Stage 5b: Local-lib prescan

[`src/prescan/local-lib.ts`](../src/prescan/local-lib.ts).

### Что делает

Для каждой объявленной library (shared или per-repo):

1. Сканирует **все `.ts/.tsx/.js/.jsx`** в `source.path` директории.
2. Для каждого файла:
   - Проверяет import declarations — есть ли импорт из `@beaver-ui/*`.
   - Бинарный флаг `hasBeaverImport`.
3. Для каждого экспортируемого компонента:
   - **Own export** → `beaverBackedBy = file.hasBeaverImport`.
   - **Re-export** (`export { X } from './Y'`) → trace до defining file, depth ≤ 5, cycle detection. `beaverBackedBy` берётся из defining file.

### Shared vs per-repo

Главное различие — **base directory для resolve'а `source.path`**:
- Shared: `configDir` (директория global config'а).
- Per-repo: `repoRoot`.

`run.ts` использует `resolveLibraries(libs, baseDir, scope)` для нормализации, потом `mergeLibraries(shared, perRepo)` для precedence — per-repo `libId` полностью заменяет shared (entry-level).

После prescan: `mergeRegistries(sharedRegistry, perRepoRegistry)` — per-repo поверх shared.

### Output

`LocalLibRegistry`. Используется в `collect.ts` для определения, является ли усмотренный JSX-импорт adoption (backed) или нужно дальше классифицировать.

## 7. Stage 6: классификация (Этап A + B)

### Этап A — `collect.ts` (Pass-A)

[`src/pipeline/collect.ts`](../src/pipeline/collect.ts).

Walk JSX, для каждого элемента:

1. **Member-expression** (`<NS.Sub.X/>`) — `readJsxTagName` возвращает `{ full: 'NS.Sub.X', base: 'NS' }`. Lookup по `base`. Member-resolution для local имеет `symbolOverride = 'X'` (последний сегмент).

2. **Lowercase tag** → html-native / neither / utility-heuristic (final).

3. **Capitalised, нет binding'а** → defined in-file, pending как local.

4. **Capitalised, binding есть** — порядок:

   a. **Beaver-пакет (canonicalized)** → final adoption/direct-beaver. Re-export через aggregator: `beaverPackage = sourcePackage`, `canonicalizedVia = aggregatorPackage`.

   b. **Local-library match** (по `matchPattern` глобу ИЛИ путь под `sourceAbsPath`):
      - Prescanned `beaverBackedBy = true` → final adoption/beaver-backed-wrapper.
      - Иначе → pending с `localLibKind = 'fully-custom'`.

   c. **In-repo relative** → pending как local.

   d. **Unresolved** с `./` prefix → pending как local (fallback).

   e. **Иначе** → third-party / neither / utility-heuristic (final).

**Pending усмотрения** идут в Pass-B. Финализированные сразу попадают в результаты.

### Между Pass-A и Pass-B: cross-file aggregation

[`run.ts/classifyRepo`](../src/pipeline/run.ts):

1. **Профили** строятся для каждой capitalised export'а в каждом файле через `buildProfilesForFile`. Profile содержит: jsxElementCount, htmlTags, propNames, beaverImports, etc.

2. **Barrel aliasing** (`aliasBarrelProfiles`) — fixpoint loop (depth=5):
   - Каждый pass перестраивает `profilesByFile` из ключей `profiles` Map'а (включая previously-added aliases).
   - Обрабатывает: `export { X } from './foo'`, `export { default as X } from './foo'`, `export * from './foo'`.
   - **Чейн barrel → barrel → source** сходится за 2 прохода благодаря rebuild'у на каждом проходе.

3. **Counters** для cross-file usage aggregation:
   - Считаются per (definingAbsPath, definingSymbol).
   - При записи в profile (`usageCount`, `filesUsedIn`) — **аккумулируются по object identity** (PF2.7): один профиль может быть зарегистрирован под несколькими ключами (barrel + direct), нельзя `=` — нужно `+=`.

### Этап B — `classify-pass.ts` (Pass-B)

Для каждого pending'а:

1. Lookup profile по `profileKey(definingAbsPath, definingSymbol)`.

2. Если profile найден → `classifyLocal(profile, signalContext)`:
   - `matchNeither` — §5.3 эвристики (`*Provider`, `*Container`, etc.) — **первыми**.
   - Если есть Beaver-импорты в профиле → adoption-wrapper rules:
     - `passesClassNameToBeaver` ИЛИ `wrapsBeaverWithStyled` → shadow / wraps-with-customization
     - `jsxElementCount ≥ threshold` (5) → shadow (substantial-markup)
     - Иначе → adoption / beaver-composition
   - Без Beaver-импортов → `detectSignals` + `pickLevel`.

3. Если profile НЕ найден:
   - `localLibKind === 'fully-custom'` → synthetic shadow + synthetic profile (для invariant completeness).
   - Иначе → neither / utility-heuristic.

4. **Дедуп ShadowComponentRecord** по `(profile.absPath, profile.componentName)` — НЕ по pending key (PF2.7 fix).

### Signal detectors (`src/classify/signals.ts`)

Каждый — pure function от `ComponentProfile`. Возвращает boolean.

```ts
detectSignals(profile, ctx) → ShadowSignal[]
```

`ctx` содержит пороги (`reusableLocalThreshold`, `substantialMarkupThreshold`) и whitelist primitive-имён.

### Level picker (`src/classify/levels.ts`)

Принимает profile + Set<ShadowSignal>. Возвращает `ShadowLevel | null`:
- `confirmed`: нет Beaver-импортов + primitive-like-name + substantial-markup
- `likely`: нет Beaver-импортов + (reusable-local OR multi-route)
- `possible`: иначе если хоть один сигнал
- `null`: ни одного сигнала

## 8. Stage 7: резолвер маршрутов

[`src/route/`](../src/route/). PRD §4.7.

### Под-стадии

1. **Discovery** (`discovery.ts`): AST-walk всех parsed файлов, ищем:
   - `createBrowserRouter([...])` / `createHashRouter` / `createMemoryRouter`
   - Variable declarators с `init.type === 'ArrayExpression'` и shape `[{ path, ... }]`
   - Exported variable declarations того же shape

2. **Page extraction** (`page-extract.ts`): для каждого route entry:
   - **`path`** — через `ConstantEvaluator` (`constant-eval.ts`):
     - String literal — fast path.
     - Иначе: trace identifier / member-expression / cross-file import. Depth ≤ 5, cycle detection.
     - Template literal без интерполяции — поддерживается.
     - Иначе → null + warning subcode (`dynamic-path-template-literal`, etc.).
   - **`element`** / **`Component`** / **`lazy`**:
     - JSX → `collectJsxCandidates` (BFS depth ≤ 5, member expressions включены).
     - Identifier → `resolveIdentifierCandidate`.
     - Lazy import → следование default export.
   - **`children`** — рекурсивный обход.

3. **Member chaser** (`member-chaser.ts`): после initial resolve, прогоняет `export ... from` цепочки depth ≤ 3 — находит **defining file**, а не barrel. Это даёт точное per-route binding.

4. **Import graph** (`import-graph.ts`): per-repo DAG. Edges:
   - Static `ImportDeclaration`
   - Dynamic `ImportExpression` (`import()`)
   - `ExportNamedDeclaration` / `ExportAllDeclaration` с source (re-exports тоже edges)

5. **Resolve** (`resolve.ts`): orchestrator. Для каждой route entry с resolvable page → `reachable(pageComponentFile)` через BFS. Затем для каждого файла:
   - `reachingRoutes(F) = { r.path | F ∈ reachable(r.pageComponentFile) }`
   - `∅` → `{ kind: 'unmapped' }`
   - 1 → `{ kind: 'bound', path }`
   - 2+ → `{ kind: 'shared', paths }`

6. **Application** (в `run.ts`): для каждого `UsageRecord` ищем биндинг файла → перезаписываем `route` поле. Также для каждой shadow-записи: если файл shared, добавляем сигнал `multi-route` (PF2.5).

### Edge cases (PRD §4.7.5)

- **Циклические импорты**: visited-set в BFS → DAG-инвариант.
- **Route-конфиг файл сам в reachable**: PRD §4.7.5 «файл входит в собственный reachable set» — реализовано через `addConfigSitesToReachable`.
- **Page-компонент не резолвится**: route регистрируется без `pageComponentFile`, не участвует в binding.

## 9. Aggregation + invariants

[`src/pipeline/aggregate.ts`](../src/pipeline/aggregate.ts).

### Algorithm

1. **Фильтрация по kind**: usages, shadowComponents.
2. **Global adoption (A)**: count adoption / (adoption + shadow).
3. **Per-repo (B)**: `groupBy(usages, repoId)` → каждой группе A.
4. **Beaver coverage (D)**: `groupBy(usages where category=beaver, beaverPackage)` → `{ reposUsing: Set.size, instances: count }`.
5. **Shadow landscape (C)**:
   - `byFile`: один entry per ShadowComponentRecord.
   - `byComponent`: `groupBy(shadowComponents, shadowGroupKey)` где `groupKey = sha256(name + sorted_props + jsxBucket).slice(0,16)`. Поднимает уровень group'ы к max'у входящих (confirmed > likely > possible).
6. **Per-route (E)**: фильтр по `route.kind === 'bound'`, потом `groupBy(repoId × routePath)`.
7. **Shared (E sibling)**: `groupBy(usages where route.kind === 'shared', repoId × filePath × componentName)`.

### Инварианты

См. [`docs/руководство.md`§10](руководство.md#10-инварианты).

Реализованы в `checkInvariants`. Iterates `usages` + аккумулирует violation-codes с counts. Сравнивает агрегатные суммы с реальными счётчиками — это то, что ловит PF2.7-класс багов (barrel-aliasing).

## 10. Viewer

[`src/viewer/render.ts`](../src/viewer/render.ts).

### Self-contained

Один HTML-файл. Структура:

```
<!doctype html>
<html lang="ru">
<head>
  <style>{inline CSS}</style>
</head>
<body>
  <main>{layout}</main>
  <script>
    const DATA = {inlined JSON};
    {client-side JS}
  </script>
</body>
</html>
```

**Никакого `fetch`, никакого backend'а, никакого CDN.** Открывается локально на машине оператора без интернета.

### JSON escaping

`<`, `>`, `&` экранируются в `<` / `>` / `&` чтобы JSON-payload не попал в `</script>` ловушку. U+2028 / U+2029 (line separators) тоже — некоторые браузеры на них ломаются в `<script>`.

### Client-side controller `createTable`

Reusable function `createTable(host, opts)`:
- `opts`: `{ rows, columns, pageSize, defaultSort, emptyText }`.
- `columns`: `[{ key, label, render?, sortable?, num?, ellipsize?, getSortValue? }]`.
- State: filter string, sort key+dir, page index, page size.
- Operations:
  - Filter: substring case-insensitive по rendered (stripped) tags всех колонок. Debounced 150ms.
  - Sort: цикл клика по headeru asc → desc → выкл.
  - Pagination: 50/100/250/500/«все».
- Render: in-place innerHTML, re-bind events.

## 12. Расширение

### Добавить новый warning code

1. Добавь литерал в `WARNING_CODES` в [`src/config/schema.ts`](../src/config/schema.ts).
2. Эмить в нужном месте `src/pipeline/` с осмысленным `message`.
3. Документируй в [`docs/предупреждения.md`](предупреждения.md).
4. Добавь regression-spec в `tests/pipeline.test.ts`.

### Добавить новый shadow signal

1. Добавь литерал в тип `ShadowSignal` в [`src/classify/signals.ts`](../src/classify/signals.ts).
2. Имплементируй детектор в `detectSignals`. Pure function от `ComponentProfile`.
3. Если меняет логику уровней — обнови [`src/classify/levels.ts`](../src/classify/levels.ts).
4. Добавь fixture + spec в `tests/classify.test.ts` + e2e в `tests/pipeline.test.ts`.
5. Документируй в [`docs/руководство.md`§11](руководство.md#11-классификация-как-сканер-решает-что-shadow-что-adoption).

### Добавить новую фикстуру

1. `tests/fixtures/fixture-<что-проверяем>/`:
   - `.beaver-scan.json`
   - `src/...`
2. Pipeline spec в `tests/pipeline.test.ts` — обязательно.
3. Если фикстура должна войти в smoke run — добавь в `example/repositories.json`.

### Добавить новый pipeline stage

⚠️ Архитектурное решение. Согласовать с founder'ом. PRD §4 фиксирует 8 стадий.

### Добавить новый source type для `LocalLibrary`

`local-path` — единственное в MVP. PRD §11.14 предлагает `git` / `npm` источники в v2.

Для нового source type:
1. Расширь `LocalLibrarySchema.source.type` в `src/config/schema.ts`.
2. Добавь branch в `prescanLibraries` (`src/prescan/local-lib.ts`) для нового type.
3. Если нужно clone'ить — переиспользуй `src/ops/git.ts`.
4. Тесты + fixture.
