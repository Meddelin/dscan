# Каталог предупреждений и инвариантов

Этот документ — справочник по двум разным наборам диагностик, которые сканер выдаёт после запуска:

- **`warnings.json`** — не-фатальные предупреждения (parse-failures, проблемы резолва маршрутов, превышенные пороги).
- **`aggregates.invariants.violations`** — нарушения доменных инвариантов §10.1 PRD. Если есть — exit code 3.

## Формат warning'а

```json
{
  "repoId": "consumer-app-1",
  "filePath": "src/pages/Checkout.tsx",
  "absPath": "/abs/path/to/repo/src/pages/Checkout.tsx",
  "code": "parse-failed",
  "message": "Unexpected token..."
}
```

- `repoId` — заполнен для любого per-repo предупреждения.
- `filePath` — repo-relative, forward-slashed. Стабильно между машинами. Используй в BI / кросс-хост пайплайнах.
- `absPath` — абсолютный путь на ФС оператора. Тот же файл, что `filePath`. Используй для быстрой навигации из терминала / IDE. **Игнорируй** в BI.
- `code` — машинно-читаемый код. См. таблицу ниже.
- `message` — человекочитаемое описание.

Оба пути присутствуют, когда warning привязан к конкретному файлу. Для предупреждений масштаба «весь репо» (`unresolved-dynamic-rate-exceeded`) — только `repoId`.

## Коды warnings

| Код | Стадия | Что значит |
|-----|--------|------------|
| `file-read-failed` | 2. Parse | Не удалось прочитать файл с диска (права, кодировка, I/O). |
| `parse-failed` | 2. Parse | `@typescript-eslint/typescript-estree` отверг исходник. Файл пропускается, скан продолжается (PRD §4.2 tolerant mode). Парсер получает `jsx: true` **только** для `.tsx`/`.jsx` — `.ts` файлы парсятся без JSX, чтобы дженерики (`<T>`) не были ошибочно прочитаны как JSX-теги. |
| `local-lib-prescan-failed` | 5b. Prescan | Не удалось просканировать `source.path` библиотеки (отсутствует на диске, массовые parse-fail'ы). Компоненты из неё фолбэкаются на per-repo `kind` (PRD §4.5). |
| `unresolved-dynamic-rate-exceeded` | 6. Classify | Доля `UnresolvedRecord` в репо превысила `thresholds.unresolvedDynamicWarningPct` (по умолчанию 5%, §5.4). Скорее всего, динамические `React.createElement` / lookup'и по строке искажают метрики. |
| `route-resolution-warning` | 7. Routes | Route entry не удалось полностью привязать к page-компоненту. `filePath` / `absPath` указывают на файл route-конфига; сообщение содержит **sub-код в квадратных скобках** (см. ниже). Route всё равно регистрируется, но не служит якорем для import-graph binding'а. |
| `repo-clone-failed` | Ops | SSH-клон консьюмер-репо упал (сеть, недоступный репо, авторизация). Скан падает fail-fast (PRD §8.3). |
| `stage7-disabled` | Ops | Stage 7 явно отключен в конфиге; usages остаются с `kind: 'unsupported'`. |

### Sub-коды `route-resolution-warning`

**Side: JSX (`element`)**

| Sub-код | Значение |
|---------|----------|
| `page-not-imported:Foo` | Identifier `Foo` использован в element но не импортирован в файле. |
| `page-import-not-in-repo:Foo from ./bar` | Foo импортирован, но `./bar` резолвится вне репо (external package). |
| `jsx-member-base-not-imported:Pages (in tag <Pages.Foo/>)` | Member-expression `<Pages.Foo/>` — base `Pages` не импортирован. |
| `jsx-member-unresolved:Pages.Foo (base Pages from ./pages)` | `Pages` импортирован, но `./pages` — external либо `Foo` не достижим через chain (PF2.6 depth=3). |
| `page-unresolved:conditional-element` | `element: cond ? <A/> : <B/>` — conditional не раскрывается (branch enumeration на element — v2). |
| `page-unresolved:hoc-wrapped-element` | `element: withSomething(<Foo/>)` — HOC от внешнего пакета. |
| `page-unresolved:non-function-lazy` | `lazy: notAFunction`. |
| `page-unresolved:no-import-in-lazy` | `lazy: () => something()` — функция, но не `import()`. |
| `page-unresolved:dynamic-import-source` | `lazy: () => import(variable)` — динамическое имя модуля. |
| `lazy-import-not-in-repo` | `lazy: () => import('@ext/foo')` — резолвится вне репо. |
| `no-jsx-candidate-resolved` | BFS по JSX-дереву (depth 5) не нашёл ни одного in-repo кандидата. |

**Side: path**

| Sub-код | Значение |
|---------|----------|
| `dynamic-path-skipped` | Generic placeholder для нераспознанной формы. |
| `dynamic-path-template-literal` | `` path: `${BASE}/x` `` — template literal с интерполяцией. |
| `dynamic-path-binding-not-found:Foo` | Identifier `Foo` использован в path, но не объявлен. |
| `dynamic-path-import-not-in-repo:Foo from ./bar` | `Foo` импортирован, но `./bar` external. |
| `dynamic-path-export-not-found:Foo in <file>` | `Foo` импортирован из `<file>`, но `<file>` не экспортирует `Foo`. |
| `dynamic-path-non-static-member` | `PATHS[key]` — computed property access. |
| `dynamic-path-cycle` | Цепочка зашла в цикл (через visited-set). |
| `dynamic-path-depth-exceeded` | Цепочка > 5 hops. |
| `dynamic-path-non-object-segment:<type>` | Path-сегмент указывает на не-object (например, `ROUTER_PATHS.foo.bar` где `foo` — string). |
| `dynamic-path-missing-property:foo` | `ROUTER_PATHS.foo` — object найден, но свойства `foo` в нём нет. |
| `dynamic-path-source-unparseable:<file>` | Cross-file resolve — целевой файл не парсится. |
| `dynamic-path-unsupported:<NodeType>` | Path-выражение — нестандартный AST-узел (CallExpression, etc.). |

Wrapper-around-page (`<Guard><Page/></Guard>`) и member-expression element (`<Pages.Foo/>`, `<NS.Sub.X/>`) **разворачиваются автоматически** — BFS по JSX-дереву (depth ≤ 5) пробует все JSX-теги в дереве, побеждает **первый**, резолвящийся в репо. Warning эмитится только когда **ни один** кандидат не подошёл.

## Добавление нового warning code

1. Добавь строковый литерал в `WARNING_CODES` в [`src/config/schema.ts`](../src/config/schema.ts).
2. Эмить в нужном месте `src/pipeline/` с осмысленным `message`.
3. Документируй в этом файле.
4. Regression-spec в `tests/pipeline.test.ts`.

## Инварианты (отдельно от warnings)

Доменные инварианты PRD §10.1 — **не** записи в `warnings.json`. Они живут в `aggregates.json` под `invariants.violations[]`:

```json
{
  "invariants": {
    "checked": 12345,
    "failed": 0,
    "violations": [
      { "code": "dataset-completeness-shadow", "count": 1, "message": "…" }
    ]
  }
}
```

Непустой `violations[]` → CLI exit code `3`, если не передан флаг `--no-fail-on-invariant`.

### Что проверяется сейчас (7 из 9)

| Код инварианта | Что гарантирует |
|----------------|-----------------|
| `shadow-level-consistency` | `bucket=shadow` ⇔ `shadowLevel` присутствует (PRD §10.1 #3). |
| `beaver-package-canonicalization` | `category=beaver` требует `beaverPackage` (§10.1 #4). |
| `route-presence` | Каждое usage имеет `route` (§10.1 #7). |
| `schema-version-fixed` | Все записи с одним `schemaVersion` (§10.1 #6). |
| `bucket-mutual-exclusivity` | Один валидный bucket на usage (§10.1 #1). |
| `classification-source-enum` | `classificationSource` — известный код (§10.1 #2). |
| `dataset-completeness-beaver` | Сумма `beaverCoverage.instances` равна числу `category=beaver` usages (§10.1 #5). |
| `dataset-completeness-shadow` | Сумма `shadowLandscape.byFile.usageCount` равна числу `bucket=shadow` usages (§10.1 #5). **Самый chrome-prone** — раньше срабатывал из-за barrel-aliasing / `export *` / default-import багов. Закрыто в PF2.7 (`bd0c517`). Если возникает на актуальной версии — пришли `warnings.json` + текст нарушения, скорее всего пятый паттерн не покрыт. |

### Отложено

- #8 per-route denominator correctness — гарантируется конструкцией `computePerRouteAdoption` (только `route.kind === 'bound'` идёт в знаменатель).
- #9 determinism hash — проверяется vitest-спецификой (`describe 'determinism (§8.4)'`), не runtime'ом скана.

## Как читать `aggregates.invariants` в HTML viewer'е

Секция «Предупреждения и инварианты» внизу отчёта свернута по умолчанию. Кнопка `Показать (N)` разворачивает блок:

- Сверху — список violation-row'ов (если есть).
- Снизу — таблица warnings с filter/sort/pagination.

Каждый violation-row показывает `<code> × <count> — <message>`. Полная история — в `aggregates.json#invariants.violations`.
